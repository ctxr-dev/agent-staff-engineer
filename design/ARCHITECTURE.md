# Architecture

How the pieces of agent-staff-engineer compose into a working system. This is the single authoritative architecture reference for the agent bundle.

Key invariants the rest of this document relies on:

- The agent does not read or write any target project's `daily/` or `knowledge/` folders. Those systems are owned by the project's own hooks and are out of scope.
- Working artefacts live under `<project>/.development/` (dot-prefix) with three subtrees: `shared/` commits with the project (team-visible reports, runbooks, configs), while `local/` and `cache/` are per-user and gitignored by default. Reusable templates stay inside the bundle.
- Bootstrap is an **interactive interview**, not a silent inspector.
- GitHub observation spans potentially multiple dev projects, multiple release projects, and extra watched repos, each with its own depth setting.
- PR merge and dev-issue Done are human gates. The agent never performs either on the human's behalf.

Every layer only reads down. Skills read templates and rules. Scripts read the schema and the config. The contract sits at the bottom because changing it cascades to everything above.

## The contract

`ops.config.schema.json` declares the shape. `ops.config.json` (one per target project) fills it. Skills read keys; they never look at the filesystem, git, or gh for project-specific values that the contract already covers.

Key groups:

- `project`: name, org, repo, default_branch, principals.
- `github.dev_projects[]`: owner, number, status_values, fields.
- `github.release_projects[]`: same shape as dev_projects[].
- `labels`: type, area, priority, intent, size, automation, state_modifiers.
- `workflow`: phase_term, branch_patterns, commits, pr, release.
- `paths`: plans_root, plan_states, done_pattern, dev_working_dir (default `.development`) with three subtrees set by `dev_working_{shared,local,cache}_subdir` (defaults `shared`/`local`/`cache`; `local/` and `cache/` are gitignored by default, `shared/` commits), templates, reports (default `.development/shared/reports`), runbooks (default `.development/shared/runbooks`). No keys touch `daily/` or `knowledge/`.
- `stack`: language, testing, platform tag arrays. Controls which memory seeds install and which test commands the dev-loop runs.
- `area_keywords`: map from area label to keyword arrays.

## Skills: responsibilities and handoffs

```text
                                 +----------------+
                                 |   user signal  |
                                 +----------------+
                                    |     |     |
                  fresh project     |     |     |   project-shape change
                  ----------------->+     |     +----------------------->
                                          |
                                    work on issue
                                          v
+----------------------+         +--------------------+        +---------------------+
| bootstrap-ops-config |         |     dev-loop       |        |    adapt-system     |
| one-time per project |         |  per dev issue     |        |  on user intent     |
+----------+-----------+         +-----+----------+---+        +----------+----------+
           |                           |          |                       |
           | writes                    | asks     | asks                  | cascades to
           v                           v          v                       v
+----------------------+         +-----------+  +---------+       +-----------------+
| ops.config.json      |<--------+ github-   |  | plan-   |       | config, labels, |
| schema-validated     |  reads  | sync      |  | keeper  |       | templates,      |
+----------------------+         +-----+-----+  +----+----+       | rules, seeds    |
                                       |             |            +-----------------+
                                       | invoked by  |
                                       v             v
                                 +-----------------------+
                                 |   release-tracker     |
                                 |   regression-handler  |
                                 +-----------------------+
```

**bootstrap-ops-config**: inspects git, gh, codebase. Produces `ops.config.json`. Hands off to `install.mjs`. No other skill runs before this one on a fresh project.

**adapt-system**: on user intent like "this is a PHI product now", produces a cascading diff across config, labels, templates, rules, seeds. Calls `github-sync` to propose label relabel plan if label taxonomy changes. Idempotent.

**github-sync**: the only skill that writes to GitHub. Hosts: label reconcile, field reconcile, issue create/update, project item add/move, rollout-to-issues converter. Every other skill that needs GitHub state goes through `github-sync`.

**dev-loop**: owns the PR lifecycle from branch to `In review`. Calls `github-sync` to move the issue status and to comment on the PR. Calls `plan-keeper` to update the plan one-liner on gate crossings. Never merges, never sets Done.

**release-tracker**: computes umbrella status from linked dev issues. Reads via `github-sync`. Auto-moves the Release umbrella status. Never touches dev issues.

**regression-handler**: triggered when the user reports a bug. Runs the lookup order (referenced commit/file, area label match, title keyword), asks `github-sync` to reopen or link issues, fills `templates/regression-report.md`, attaches to the issue.

**plan-keeper**: enforces folder placement, checkbox format, frontmatter, lifecycle moves (todo -> in-progress -> in-review -> done). Invokes `github-sync` only to confirm the linked issue status on a plan move. Does not write to `daily/` or `knowledge/`. Any project that already runs a daily-log or knowledge-base system keeps running it through its own hooks; the agent stays out of the way.

## Code-review provider

`dev-loop` does not roll its own review engine. Before opening a PR, it delegates the self-review step to the provider named in `ops.config.json -> workflow.code_review.provider`.

Default: [`ctxr-dev/skill-code-review`](https://github.com/ctxr-dev/skill-code-review). Multi-specialist skill, up to 18 reviewers in parallel, stack-aware overlays, integrated linters, verdict in the set `GO / CONDITIONAL / NO-GO`.

```text
   dev-loop
      |
      | self_review step
      v
   +----------------------------+
   | read workflow.code_review  |
   +------+---------------------+
          |
   +------+-----------+-----------+
   v                  v           v
 provider=           provider=   provider=
 ctxr-skill-         internal-   none
 code-review         template
   |                   |           |
   v                   v           v
 invoke              render      skip self-review
 /skill-code-        templates/  (only allowed if
 review              code-       workflow.pr.
 (args from          review-     self_review_
 mode, output_       report.md   required is
 format, scope)      and prompt  false)
   |                 the author
   v
 block on
 verdict in
 block_on_verdict
```

The agent does not ship `ctxr-dev/skill-code-review`. At install time, dev-loop detects whether the skill is available; if the provider is `ctxr-skill-code-review` and the skill is missing, it prints the configured `install_hint` and asks whether to fall back to the internal template or halt. No silent degradation.

## Dev-loop state machine

```text
       [issue: Backlog or Ready]
                  |
                  |  user picks issue, runs dev-loop
                  v
       [branch from default_branch]
                  |
                  v
       [edit / implement]
                  |
                  v
       [local review loop]
         format -> lint -> type -> unit -> e2e
                  |
                  |  any fail -> back to edit
                  v
       [self-review report filed]
         at <project>/.development/shared/reports/
         (committed; shared/ subtree is team-visible)
                  |
                  v
       [PR opened, reviewers requested]
                  |
                  |--> github-sync: issue -> In review
                  |--> plan-keeper: mark [x] in plan
                  |--> release-tracker: recompute umbrella
                  v
       [address review comments loop]
                  |
                  |  each push -> status stays In review
                  v
       ***  HUMAN GATE: merge PR  ***
                  |
                  v
       [PR merged]
                  |
                  |--> github-sync: keep issue at In review
                  |--> plan-keeper: update plan (no-op if already [x])
                  v
       ***  HUMAN GATE: mark issue Done  ***
                  |
                  v
       [issue: Done]
                  |
                  |--> release-tracker: recompute umbrella
                  v
           [dev-loop exits]
```

Two explicit human gates. `dev-loop` never transitions past either.

## Release-tracker computation

For each Release umbrella:

```text
linked_dev_issues = gh api <release item> .sub-issues
if every issue.status == Done:
    umbrella.status = Done
elif any issue.status in {Ready, In progress, In review}:
    umbrella.status = In progress
    umbrella.fields.Linked Dev Issues = count(linked_dev_issues)
    umbrella.fields.blocker_count = count(issues with label=blocked)
else:
    umbrella.status = Backlog
```

Triggered on any of:

- Dev issue status change (via webhook or polled check).
- New issue linked to a release umbrella.
- Issue removed from umbrella.

`release-tracker` writes only to the release project. Never mutates the dev project.

## Wrapper model

Canonical skills, rules, memory seeds, and templates live **only** inside the agent bundle at `paths.agent_bundle_dir` (default `.claude/agents/agent-staff-engineer/`). The bundle is its own git repo; `git pull` inside that folder updates the canonical content.

At install time, `install.mjs` does not copy those files out. It generates **wrappers** at the canonical Claude Code locations:

- `paths.wrappers.skills_dir` (default `.claude/skills/`): one `<agent>_<name>/SKILL.md` wrapper per canonical skill. The `<agent>` prefix is derived from the bundle's `package.json -> name` (scope stripped) via `scripts/lib/agentName.mjs` so wrappers cannot collide with files shipped by other agents or skills.
- `paths.wrappers.rules_dir` (default `.claude/rules/`): one `<agent>_<name>.md` wrapper per canonical rule. Same prefix rule.
- The target's Claude memory folder: one wrapper memory entry per seed at `seed-<agent>_<name>.md` for seeds matching the target's stack tags.
- `<target>/CLAUDE.md`: a project-level managed-block injection that points at the bundle's rules and the ops config. If the file already exists, only the block between the two managed-block markers is owned by the installer; everything outside is preserved byte-for-byte. Implemented by `scripts/lib/inject.mjs`.

### Wrapper anatomy

Every wrapper has the same three-part shape:

```markdown
---
<frontmatter declaring name, description, etc. so Claude Code registers it>
---

<!-- Wrapper file. Canonical content lives at <bundle-path>. Updates via
     'git pull' inside the bundle. Edit below the overrides marker only. -->

Before acting, read and follow the canonical version at:
`<bundle-path>/<relative>.md`

That file is updated by `git pull` inside the bundle; this wrapper continues
to point at the same path so changes are picked up automatically.

<!-- ============ PROJECT OVERRIDES BELOW (preserved across agent updates) ============ -->

<!-- Add project-specific overrides here. Installer runs never touch anything
     below the marker. -->
```

### Regeneration rules

- **On fresh install (`--apply`)**: generate every wrapper; below-marker section is empty.
- **On update (`--update`)**: for each existing wrapper, parse the marker; rewrite everything **above** it from current bundle state (refreshed frontmatter, header notice, include instruction, path). Preserve everything **below** the marker byte-for-byte.
- **On new canonical files between updates**: create fresh wrappers for them. Existing wrappers untouched.
- **On removed canonical files between updates**: leave the orphaned wrapper in place with a one-line warning inserted above the marker. User decides to keep it (their overrides may still be valuable) or delete it.
- **On `--uninstall`**: remove every wrapper listed in `.install-manifest.json`. Warn before removing any wrapper whose below-marker section contains user content.

### Why wrappers and not copies or symlinks

- **Copies**: lose the auto-update story. Every `git pull` would need to re-run the installer to refresh `.claude/skills/*`.
- **Symlinks**: auto-update works, but there is no place to add project-specific overrides.
- **Wrappers**: give us auto-update (paths stay stable, `git pull` refreshes canonical content behind the include) AND a per-wrapper override surface that survives updates.

## Installer flow (one pass)

```text
install.mjs --target T --apply
  |
  +-- preflight(): Node >= 20, gh available, gh auth, git available
  |
  +-- bundle_schema_check(): every key consumed by bundle files exists in ops.config.schema.json
  |
  +-- if T/<paths.agent_bundle_dir>/ missing: error (user forgot the git clone step)
  |
  +-- enumerate_canonical():
  |     walk bundle's skills/, rules/, memory-seeds/ (filter by stack tags)
  |     produce { name, bundle_path, target_wrapper_path } per item
  |
  +-- bootstrap(): interactive interview + propose ops.config.json, user confirm
  |     writes <T>/.claude/ops.config.json  (outside the bundle, never inside)
  |     writes <T>/.claude/agents/agent-staff-engineer/.bootstrap-answers.json (inside the bundle, gitignored)
  |
  +-- render(CLAUDE.md) from the inline CLAUDE.md template in scripts/install.mjs at <T>/CLAUDE.md
  |    (also a wrapper pointing at bundle rules)
  |
  +-- ensure_working_dir(): create T/.development/ + shared/ + local/ + cache/ if missing
  |     write shared/README.md first-time so team knows what commits where
  |     add local/ + cache/ paths to T/.gitignore if not already present
  |     (never ignore .development/ itself so shared/ can commit)
  |
  +-- generate_skill_wrappers():
  |     for each canonical skill, write <paths.wrappers.skills_dir>/<agent>_<name>/SKILL.md
  |     (agent prefix derived from package.json; prevents filename collisions)
  |     above marker: frontmatter + header notice + include instruction + bundle path
  |     below marker: empty
  |
  +-- generate_rule_wrappers():
  |     for each canonical rule (excluding product-*.md), write <paths.wrappers.rules_dir>/<agent>_<name>.md
  |     same three-part shape as skills
  |
  +-- install_memory_seeds(): filter seeds by stack tags; for each applicable seed
  |     write a wrapper memory entry at seed-<agent>_<name>.md in the target's Claude memory folder
  |     (frontmatter name description type, include directive, marker, empty overrides)
  |     |
  +-- write .install-manifest.json inside the bundle:
  |     every wrapper path + its canonical source + content hash
  |     used by --update (hash drift) and --uninstall (rollback)
  |
  +-- summary_report(): wrappers created, stack-filtered seeds, CLAUDE.md injection result
```

## Uninstall flow

```text
install.mjs --target T --uninstall
  |
  +-- read .install-manifest.json
  |
  +-- for each listed wrapper:
  |     parse at marker
  |     if below-marker content is non-empty:
  |       warn, surface the full below-marker text, ask the user whether to
  |       preserve the file (rename .userkeep.md) or delete
  |     else: delete the wrapper
  |
  +-- CLAUDE.md: strip the managed block between the begin/end markers.
  |     if nothing non-whitespace remains OUTSIDE the block, delete the file;
  |     otherwise leave the file in place without the block (preserves every
  |     byte of user-authored content outside the block).
  |
  +-- leave untouched:
  |     ops.config.json (user owns it)
  |     any rules/product-*.md (project-specific, user owns)
  |     any .development/shared/ content (committed; user/team owns)
  |     any .development/local/ and .development/cache/ content (per-user)
  |     memory wrapper entries whose below-marker sections carry user text
  |
  +-- remove .install-manifest.json
  |
  +-- summary: "bundle folder can be removed with 'rm -rf <paths.agent_bundle_dir>'; your below-marker overrides that were preserved are listed above."
```

The user's project-owned assets survive. Re-install later picks them up as pre-existing state; the installer detects wrappers whose below-marker sections are already populated and re-links them to the new canonical sources.

## Update flow

```text
install.mjs --target T --update
  |
  +-- preflight
  |
  +-- re-run bundle_schema_check for the current bundle (post 'git pull')
  |
  +-- if ops.config.schema.json grew new required keys:
  |     run adapt-system internally to propose values
  |     user confirms; write the diff to ops.config.json
  |
  +-- for each entry in .install-manifest.json:
  |     read the existing wrapper
  |     split at the marker (paths.wrappers.marker)
  |     rewrite everything ABOVE the marker from current bundle state
  |       (frontmatter may change if canonical frontmatter changed,
  |        header notice refreshed, include path refreshed)
  |     preserve everything BELOW the marker byte-for-byte
  |     update manifest hash
  |
  +-- enumerate canonical files again:
  |     for any canonical file that lacks a wrapper (new skill/rule/seed), create one
  |     for any wrapper that now points at a missing canonical file, insert a
  |       one-line warning above the marker; do not delete the wrapper
  |
  +-- summary: wrappers refreshed, wrappers added, wrappers flagged orphaned
```

Update never overwrites `ops.config.json`, project rules, below-marker wrapper content, or anything under `.development/`. It refreshes only the above-marker portion of each managed wrapper plus any new wrappers.

Content updates that do not change the set of canonical files (the common case) need no `install.mjs --update` run at all: `git pull` inside the bundle is sufficient, because wrappers reference in-bundle paths and pick up the new content on the next Claude session. Re-running `--update` becomes necessary only when (a) the bundle's canonical file set changes, (b) the schema grows required keys, or (c) the user wants the wrapper headers refreshed.

## Multi-target GitHub observation

The agent reads and writes GitHub according to a list of observed targets declared in `ops.config.json`. Each entry carries a depth setting that controls what the agent is allowed to do there.

```text
+-------------------------+
|   ops.config.json       |
|   github:               |
|     dev_projects: [...] |
|     release_projects:   |
|     observed_repos:     |
+------------+------------+
             |
             v
+-------------------------+        +---------------------+
|   github-sync           |<------>|   gh CLI            |
|   per-target depth      |        |   (read + write)    |
|   gates every call      |        +---------------------+
+------------+------------+
             |
     +-------+-------+----------------+
     v               v                v
+---------+    +----------+   +-----------+
| dev-    |    | release- |   | regression|
| loop    |    | tracker  |   | -handler  |
+---------+    +----------+   +-----------+
```

Every skill that hits GitHub goes through `github-sync`. `github-sync` checks the target's `depth` before any write, and refuses writes on `read-only` or out-of-scope items.

Examples:

- A target of `depth: umbrella-only` lets `release-tracker` update umbrella status but rejects any dev-issue write.
- A target of `depth: labeled:cross-repo` lets `regression-handler` open issues labelled `cross-repo` in a watched repo but rejects unrelated writes.
- A target of `depth: read-only` lets Claude read and cite without ever posting.

## What the agent deliberately does NOT touch

The agent stays out of these systems even if they exist in the target project:

- `<project>/daily/` or any similar daily-log folder. Owned by the project's hooks or ops.
- `<project>/knowledge/` or any Obsidian, Dendron, or similar knowledge base.
- `<project>/docs/` unless a template or rule explicitly references a doc path in `ops.config.json -> paths`.
- Anything under `<project>/.obsidian/`, `.vault/`, or similar private-tool config.
- The project's existing Claude memory folder at `~/.claude/projects/<path>/memory/`. The agent ships `memory-seeds/` as starter content only; it never rewrites existing entries.

Any cross-cutting log that the user wants to keep up to date is their job, through their own hooks. The agent's sphere ends at GitHub, the bundle, `<project>/.development/`, the plans folder, and `ops.config.json`.

## Zero-deps enforcement

Every `.mjs` file imports only from:

- Node.js built-ins (`node:fs/promises`, `node:child_process`, `node:path`, `node:url`, `node:util`, `node:process`, `node:readline/promises`).
- Relative paths into `scripts/lib/` or sibling `scripts/*.mjs`.

`validate_bundle.mjs` greps every `.mjs` for bare imports and fails if any import specifier does not start with `node:`, `.`, or `/`. No `package.json` may exist inside `scripts/`.

## Failure modes per skill

| Skill | Common failure | Behavior |
| --- | --- | --- |
| bootstrap-ops-config | gh not authed | prints `gh auth login` guidance, exits |
| bootstrap-ops-config | no git remote | prints error, exits, asks user to initialise |
| adapt-system | intent ambiguous | asks clarifying question via AskUserQuestion, does not guess |
| github-sync | rate-limited | retries with backoff; if still failing, surfaces and halts |
| dev-loop | tests failing locally | halts at the failed stage, does not push |
| dev-loop | PR template missing sections | halts, asks user to fill |
| release-tracker | sub-issue link broken | logs warning, continues with remaining |
| regression-handler | no match in lookup | opens a new bug issue; logs that nothing was reopened |
| plan-keeper | plan file format invalid | reports exact violation, refuses to move |
